## 1. SSH fallback to open the ES port  ✅ (implemented + fixed)

**Goal:** if the ES port (default 9200) is closed on the CC machine, SSH in and
open it via the firewall — **only after a direct ES connection fails**.

### Backend
- **`services/ssh_opener.py`** (new):
  - `build_open_port_command(ssh_user, port)` — chooses the command by user, **port is unquoted**:
    - `root`  → `/opt/radware/box/bin/net_firewall_open-port.sh set <port> open`
    - other   → `net firewall open-port set <port> open`
  - `open_port_via_ssh(...)` — connects via **paramiko**, runs the command through an
    **interactive shell (`invoke_shell`, allocates a PTY)** because the CC login shell is
    the Radware restricted **`pshell`**, which rejects `exec_command`
    (error was: `pshell: can't access tty; job control turned off`).
  - `_run_in_interactive_shell()` — drives the PTY: drain banner → send command → drain output → `exit`.
  - Success is confirmed by re-checking **TCP reachability** of the port (`port_is_open`, polls ~6s)
    plus scanning output for error keywords.
- **`routers/query.py` → `/api/connect`**: `ConnectionRequest` gained `ssh_enabled`,
  `ssh_user`, `ssh_password`, `ssh_port` (default 22). Flow: try direct ES → on fail, if
  `ssh_enabled` run `open_port_via_ssh` → retry ES. Returns `ssh_opened_port` + `ssh_command`
  on success, or a clear error.
- **`requirements.txt`**: added `paramiko>=3.0.0`.

### Frontend
- **`index.html`**: SSH section in the connection card — "Open ES port via SSH if closed"
  checkbox (`connSshEnabled`) + `connSshUser` / `connSshPass` (eye toggle) / `connSshPort`
  fields inside `#sshFields` (hidden until enabled).
- **`app.js`**: `readForm`/`fillForm`/`clearForm` include SSH fields; `toggleSshFields()`,
  `toggleSshPassVis()`; connect feedback shows the executed SSH command when the port was
  opened over SSH. SSH fields persist in `localStorage` (like the ES password) so silent
  reconnect works — **note: SSH password is stored in LS**, flag if that's undesirable.

## 2. Table cell controls anchored on hover  ✅

Problem: cells with very long text pushed the edit/delete controls off-screen.
- **`style.css`**: `.rt-cellctrl` is now **absolutely positioned, pinned to the right edge**,
  vertically centered, as a floating white chip (`z-index:4`). Appears on `.rt-cell:hover`.
- Long text: `.rt-val` truncates with ellipsis (`max-width:340px; white-space:nowrap`).
- **`app.js`**: each value span has `title="<full value>"` for the full-text tooltip.

## 3. Pretty-JSON view + edit in table cells  ✅

- **`app.js`**:
  - `asJsonObject(v)` — returns parsed object/array if the value is (or JSON-encodes) one.
  - Table cell rendering (`buildInteractiveTableHtml`): for JSON cells, a **`{ }` braces
    button** is added to the hover chip (row index `ri` passed → `showJsonCell(ri, field, this)`).
    The braces button shows **even in read-only mode**; edit/delete pencils/trash stay gated on write mode.
  - `showJsonCell(ri, field, el)` → looks up the row via `tableDisplayRows()[ri]`, calls
    `showJsonModal(doc, {title, value, id, index, field})`.
  - `showJsonModal(doc, opts)` — pretty-prints (2-space). **View mode**: Copy + (if editable) Edit + Close.
    **Edit mode** (only when `writeMode` && row has `_id`/`_index`): a dark **textarea** with
    Format / Save / Cancel. Save parses JSON (inline red error if invalid) then calls
    `applyDocChange(id, index, field, 'set', parsedObject)`. Escape/outside-click disabled while editing.
  - Both `showJsonCell` and `showJsonModal` are bound into the **pop-out** window.
- **Backend**: reuses `POST /api/doc/update` (`value: Any`; `_coerce_value` passes objects
  through unchanged → nested structure preserved, not stringified).
- **`style.css`**: `.rt-modal-json` (wider modal), `.rt-json-pre` (view), `.rt-json-edit` (editor textarea).
